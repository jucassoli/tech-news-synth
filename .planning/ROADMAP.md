# tech-news-synth — Roadmap

**Milestone:** v1 (initial production-ready agent on @ByteRelevant)
**Granularity:** standard (8 phases)
**Last updated:** 2026-04-12

## Core Value

Transformar ruído de feeds de tecnologia em um post por ciclo que destaca o tema com mais cobertura e o ângulo único de cada fonte — sem repetir o mesmo assunto em 48h.

## Phases

- [ ] **Phase 1: Foundations** — Docker + config + structured logging + scheduler skeleton running a no-op cycle
- [ ] **Phase 2: Storage Layer** — Alembic migrations + 4 tables (articles/clusters/posts/run_log) with idempotent repos
- [ ] **Phase 3: Validation Gate** — Live smoke tests of Anthropic + X OAuth + real post cost, documented before pipeline build-out
- [ ] **Phase 4: Ingestion** — Configurable sources, per-source failure isolation, conditional GET, normalized `Article` stream
- [ ] **Phase 5: Cluster + Rank** — Pure-core TF-IDF clustering, winner selection, 48h centroid anti-repeat, fallback picker
- [ ] **Phase 6: Synthesis** — Claude Haiku 4.5 PT-BR synthesis with grounding + weighted char-budget enforcement
- [ ] **Phase 7: Publish** — Idempotent X posting with daily/monthly caps and rate-limit handling
- [ ] **Phase 8: End-to-End + Hardening** — Full pipeline wired, 48h dry-run soak, operator CLIs, live cutover

## Phase Details

### Phase 1: Foundations
**Goal**: A running, observable, scheduled container with validated secrets and UTC invariants — the chassis every later phase writes through
**Depends on**: Nothing (first phase)
**Requirements**: INFRA-01, INFRA-02, INFRA-03, INFRA-04, INFRA-05, INFRA-06, INFRA-07, INFRA-08, INFRA-09, INFRA-10
**Success Criteria** (what must be TRUE):
  1. Operator runs `docker compose up` on a clean host and sees an `app` container plus a `postgres` container, both healthy, with named volumes for DB data and logs
  2. Boot fails fast with a clear error if any required `.env` key is missing or malformed (pydantic-settings validation); `.env.example` is present and `.env` is gitignored
  3. APScheduler (PID 1, UTC) fires a no-op `run_cycle()` every 2h (configurable via `INTERVAL_HOURS`); one JSON log line per cycle appears on stdout AND on the logs volume, each tagged with a unique `cycle_id`
  4. Raising an exception inside `run_cycle()` is logged with a full stacktrace but the scheduler keeps ticking — no crash loop
  5. Setting `PAUSED=1` or touching `/data/paused` causes the next cycle to exit 0 with a log line and zero I/O; setting `DRY_RUN=1` is accepted by config and visible in the cycle log

### Phase 2: Storage Layer
**Goal**: A versioned schema that every downstream feature (dedup, 48h window, audit, replay, daily cap) writes through, with idempotent upserts proven by tests
**Depends on**: Phase 1
**Requirements**: STORE-01, STORE-02, STORE-03, STORE-04, STORE-05, STORE-06
**Success Criteria** (what must be TRUE):
  1. `alembic upgrade head` runs automatically on container startup and brings a fresh Postgres to the current schema; downgrade path exists for at least the latest migration
  2. The four tables — `articles`, `clusters`, `posts`, `run_log` — exist with the documented columns; every timestamp column is `TIMESTAMPTZ`
  3. Re-inserting the same article (same canonical URL) is a no-op: `article_hash` is `UNIQUE`, upsert uses `ON CONFLICT DO NOTHING`, and a unit test proves a rerun of the same batch leaves row count unchanged
  4. A `posts` row can be written with `status` in `{pending, posted, failed, dry_run}`, a `theme_centroid` payload, `tweet_id`, `cost_usd`, and distinct `created_at` / `posted_at` timestamps
  5. `run_log` captures every cycle with `cycle_id`, start/finish times, status, per-source fetch counts, and cluster counts — queryable for the last 14+ days of articles (retention covers 48h anti-repeat with margin)

### Phase 4: Ingestion
**Goal**: A robust, configurable fetch layer that produces a normalized `Article` stream and never lets one bad feed abort a cycle
**Depends on**: Phase 2 (writes to `articles`), Phase 3 (GO decision)
**Requirements**: INGEST-01, INGEST-02, INGEST-03, INGEST-04, INGEST-05, INGEST-06, INGEST-07
**Success Criteria** (what must be TRUE):
  1. `sources.yaml` is mounted into the container and drives the active source set; adding, editing, or removing a source requires no code change; a malformed entry fails boot with a clear error pointing to the bad line
  2. A cycle run fetches from all five v1 sources (TechCrunch RSS, The Verge RSS, Ars Technica RSS, Hacker News Firebase, Reddit r/technology JSON) with the descriptive `ByteRelevant/0.1` User-Agent and per-source timeout + `tenacity` retry (max 3, exponential backoff)
  3. Simulating a 5xx or timeout on one source logs a structured warning, skips that source, and lets the cycle complete with the remaining sources' articles upserted to the DB
  4. A second cycle against unchanged feeds sends `If-None-Match` / `If-Modified-Since` from stored ETag/Last-Modified and processes zero new rows (conditional GET works)
  5. Each persisted article exposes `id`, `source`, `url`, `canonical_url`, `title`, HTML-stripped `summary`, UTC-aware `published_at`, `fetched_at`, and `article_hash`; a source with 20 consecutive failures is auto-disabled and re-enabled only via CLI

### Phase 5: Cluster + Rank
**Goal**: A deterministic, pure-core clustering + ranking module that picks the cycle's winning topic, enforces 48h semantic anti-repetition, and always has a fallback so cadence holds
**Depends on**: Phase 4 (consumes `Article` stream + real fixture data)
**Requirements**: CLUSTER-01, CLUSTER-02, CLUSTER-03, CLUSTER-04, CLUSTER-05, CLUSTER-06, CLUSTER-07
**Success Criteria** (what must be TRUE):
  1. Given the last 6h of articles (configurable `CLUSTER_WINDOW_HOURS`), the clusterer produces zero-or-more clusters using TF-IDF (char n-grams 3–5, PT+EN stopwords, `unidecode`) + cosine + agglomerative at the configured threshold (default 0.35)
  2. Winner selection is fully deterministic on fixtures: primary key = distinct source count, tiebreak 1 = most recent article, tiebreak 2 = source-weight sum; the same fixture always picks the same cluster
  3. A fixture post from <48h ago with centroid cosine ≥ 0.5 to the current winner causes the winner to be rejected and the next-best cluster selected (semantic, not lexical, anti-repeat)
  4. On a "slow news day" fixture where no cluster meets threshold, the fallback picker returns a single best-ranked article so the cycle still has something to publish (cadence > strict threshold, per Core Value)
  5. Every cycle persists a `clusters` audit trail: which cluster won, scores for all candidates, and which were rejected by the anti-repeat filter — queryable for operator inspection
**Plans:** 2 plans
Plans:

### Phase 6: Synthesis
**Goal**: Turn the winning cluster into a grounded PT-BR post that provably fits inside the X char budget, with cost and token usage logged per call
**Depends on**: Phase 5 (consumes winner cluster)
**Requirements**: SYNTH-01, SYNTH-02, SYNTH-03, SYNTH-04, SYNTH-05, SYNTH-06, SYNTH-07
**Success Criteria** (what must be TRUE):
  1. The synthesizer calls the Anthropic SDK with the pinned model id `claude-haiku-4-5` (never an alias), `max_tokens=150`, and input limited to the cluster's 3–5 article titles + short summaries (never full bodies)
  2. The prompt is PT-BR jornalístico neutro with explicit grounding guardrails ("use APENAS informações das manchetes/resumos; NÃO invente datas/nomes/citações; mantenha nomes próprios intactos")
  3. A post that exceeds `280 − 23 (t.co) − hashtag_budget` under weighted char count triggers up to 2 "encurte para N" re-prompt retries; if still over, last-resort whitespace-aware truncation with ellipsis is applied — and a human reviewing 10 fixture synthesized posts finds every one under the limit
  4. The final output string follows `<síntese PT-BR> <source-URL> <hashtag(s)>`, always includes the source URL for attribution, and uses 1–2 hashtags selected from an allowlist (not freestyled by the LLM)
  5. Every synthesis call logs `input_tokens`, `output_tokens`, and `cost_usd` in the cycle summary AND writes `cost_usd` to the corresponding `posts` row
**Plans**: 2 plans

### Phase 7: Publish
**Goal**: Idempotently post the synthesized tweet to @ByteRelevant with hard guardrails on quota, cost, and rate limits — the only irreversible side effect in the pipeline
**Depends on**: Phase 6 (consumes synthesized post), Phase 3 (validated OAuth baseline)
**Requirements**: PUBLISH-01, PUBLISH-02, PUBLISH-03, PUBLISH-04, PUBLISH-05, PUBLISH-06
**Success Criteria** (what must be TRUE):
  1. Publishing uses `tweepy.Client.create_tweet` with OAuth 1.0a User Context (4 secrets); boot explicitly rejects bearer-token-only configs with a clear error
  2. A `posts` row is inserted with `status=pending` BEFORE the X API call; on success it's updated to `posted` + `tweet_id` + `posted_at`, on failure to `failed` with error details — and a simulated mid-call crash followed by retry does NOT produce a duplicate tweet (idempotent)
  3. A 429 response reads `x-rate-limit-reset`, emits a structured warning, and skips the remainder of the cycle cleanly without crashing the scheduler
  4. Once `posts.posted_at` count for the current UTC day reaches `MAX_POSTS_PER_DAY` (default 12), the publisher short-circuits (cluster + synthesis still run and log) and the cycle exits clean; crossing `MAX_MONTHLY_COST_USD` triggers a hard kill-switch
  5. With `DRY_RUN=1`, no X API call happens; instead a `posts` row with `status=dry_run` and the full synthesized text is written for human review
