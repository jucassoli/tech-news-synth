# Phase 2: Storage Layer - Context

**Gathered:** 2026-04-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Deliver a versioned Postgres schema (`articles`, `clusters`, `posts`, `run_log`) managed by Alembic, with SQLAlchemy 2.0 typed models, `TIMESTAMPTZ` everywhere, and idempotent upserts proven by tests. Scope includes: Alembic bootstrap + first migration, SA 2.0 `Mapped[...]` models, repository helpers for each table, `ON CONFLICT DO NOTHING` upsert for `articles`, sessionmaker wired into `run_cycle()`, and `alembic upgrade head` auto-run on container startup before the scheduler begins. Out of scope: ingestion/normalization of feed data (Phase 4), clustering logic (Phase 5), actual centroid population (Phase 5/7 fill the `theme_centroid` BYTEA column), publish logic (Phase 7).

</domain>

<decisions>
## Implementation Decisions

### Migration Pipeline
- **D-01:** `alembic upgrade head` runs **in the app container's entrypoint, before `scheduler.start()`**. Implementation: extend `src/tech_news_synth/__main__.py` to call a `run_migrations()` helper (from a new `src/tech_news_synth/db/migrations.py`) after `load_settings()` and before `build_scheduler()`. On migration failure, propagate the exception so the container exits non-zero and compose surfaces the error.
- **D-02:** **Autogenerate + post-edit** migration style. `alembic revision --autogenerate -m "..."` after model changes; human reviews the generated Python and edits to:
  - force `TIMESTAMPTZ` (autogenerate emits `DateTime(timezone=True)` which is correct in SA 2.0 but verify)
  - annotate `BYTEA` (`sa.LargeBinary()`) and `JSONB` (`postgresql.JSONB`) columns
  - add partial / expression indexes if needed
  - confirm the `ON CONFLICT DO NOTHING` usage lives in upsert helpers, not in the migration
- **D-03:** **Compose `depends_on: service_healthy` is the sole DB-readiness gate.** No tenacity retry around alembic or session creation in v1. If postgres is down, compose already blocks app startup; if it flaps mid-cycle, the cycle fails gracefully via INFRA-08 exception isolation and the scheduler retries on the next tick.

### Row Identity & Hashing
- **D-04:** **`bigserial` (auto-increment `bigint`) for `articles.id`, `clusters.id`, `posts.id`.** Small (8 bytes), Postgres-native, fast foreign keys. Monotonic within each table.
- **D-05:** **`run_log.cycle_id` is the natural PK**, `TEXT` column storing the 26-char ULID already generated by Phase 1 (`tech_news_synth.ids.new_cycle_id()`). No surrogate `run_log.id`. Downstream tables that reference a cycle (`clusters.cycle_id`, `posts.cycle_id`) use `TEXT` + FK to `run_log.cycle_id`.
- **D-06:** **`article_hash = SHA256(canonical_url)`** (overrides the initial recommendation of SHA1). Stored as `CHAR(64)` with `UNIQUE` index. Canonicalization (strip utm_*, normalize scheme/host to lowercase, drop fragment, sort query params) is a pure-function helper `tech_news_synth.db.hashing.canonicalize_url(url) -> str` + `article_hash(url) -> str`. Phase 4 ingest uses it; Phase 2 only delivers the helper and test.

### Centroid Storage
- **D-07:** **`posts.theme_centroid` is `BYTEA`** holding a dense numpy `float32` vector serialized via `np.asarray(x, dtype=np.float32).tobytes()`. Read-back with `np.frombuffer(blob, dtype=np.float32)`. Column is nullable (populated by Phase 5/7, not this phase). No pgvector extension; no sparse TEXT[] representation. Anti-repetition (CLUSTER-05) will read last-48h centroids into numpy and compute cosine similarity client-side — acceptable at ~24 rows per 48h window.

### Retention
- **D-08:** **Passive floor — no active cleanup in v1.** STORE-06's "≥14 days of articles" is trivially satisfied because nothing deletes. Volume budget: ~500 articles/day × 365 days ≈ 180k rows/year — well under Postgres scale concerns. A future `ARTICLE_RETENTION_DAYS` config + scheduled cleanup is explicitly deferred; the schema does NOT carry a `retention_policy` column or any soft-delete flag.

### Claude's Discretion
- SA 2.0 `DeclarativeBase` subclass layout (one `Base` in `db/base.py` or split per module — Claude picks).
- Session lifecycle: `sessionmaker` + context manager per `run_cycle()` call; single connection per cycle. Connection pool: default SA `QueuePool` size=5 (single-worker, plenty of headroom).
- Repository helper shape — module-level functions (`articles.upsert_batch(session, rows)`) vs classes. Claude picks; prefer module-level functions for this codebase (matches pure-function Phase 1 style).
- Test DB strategy: recommend **transactional rollback fixture** (each test opens a transaction, rolls back on teardown) against the compose postgres during CI/local. For unit tests of pure helpers (canonicalize_url, article_hash), no DB needed. Claude decides between this and testcontainers-python.
- Alembic `env.py` online-migration setup (typical boilerplate; use SA 2.0 + psycopg3 DSN from settings).
- `run_log.counts` column type — `JSONB` to keep per-source fetch counts + cluster count flexible (schema may evolve in Phase 4/5). Not a locked decision — Claude may choose a narrower shape if it prefers.
- `posts.status` enum — Postgres `CHECK` constraint with `{pending, posted, failed, dry_run}` rather than a pg ENUM type (easier to evolve via migration).

</decisions>

## Existing Code Insights

### Reusable Assets (from Phase 1)
- `tech_news_synth.config.Settings.database_url` — pre-built `postgresql+psycopg://user:pw@host:port/db` DSN. Use it for the SA engine and Alembic `sqlalchemy.url`.
- `tech_news_synth.ids.new_cycle_id()` — ULID string generator; `run_log.cycle_id` FK origin.
- `tech_news_synth.logging.get_logger()` — structlog logger; DB layer should log via this (never `print`).
- `tech_news_synth.config.Settings` fields already pinned: `postgres_host`, `postgres_port`, `postgres_db`, `postgres_user`, `postgres_password` (SecretStr).

### Established Patterns
- **Pure modules + functions, no classes unless warranted** (Phase 1 style — `killswitch.is_paused()`, `ids.new_cycle_id()`).
- **Fail-fast at boot** (pydantic-settings in Phase 1). Migrations follow the same model: error propagates, container exits non-zero.
- **`datetime.now(timezone.utc)`** — never naive datetimes. `tests/unit/test_utc_invariants.py` enforces via ruff DTZ rules.
- **structlog contextvars bound at cycle start** — any DB log line inside a cycle inherits `cycle_id` + `dry_run` automatically.
- **pytest conftest fixtures** already provide `monkeypatch_env` and `tmp_data_dir`. Add `db_session` and `clean_db` fixtures for Phase 2.
- **Test layout** — `tests/unit/` mirrors `src/tech_news_synth/`. New: `tests/unit/test_db_*.py` plus `tests/integration/test_*.py` for tests that hit real postgres.

### Integration Points
- `src/tech_news_synth/__main__.py` gains a `run_migrations()` call before `build_scheduler()` — the one ordering change outside `db/`.
- `src/tech_news_synth/scheduler.py`'s `run_cycle()` needs a session factory injected (or imported from `db/session.py`) so later phases can write to `run_log` at cycle start/end. Phase 2 may wire the `run_log` INSERT + UPDATE skeleton so that Phase 4+ only fill in counts.
- `compose.yaml` already has `postgres:16-bookworm` with healthcheck. No compose changes for Phase 2 (migrations run in-app).
- Alembic config lives at repo root: `alembic.ini` + `alembic/` (env.py + versions/). Docker image copies both into `/app`.

<specifics>
## Specific Ideas

- Initial migration is one file creating all four tables with every column + constraint + unique index. Naming: `alembic/versions/<rev>_initial_schema.py` with message `"initial schema: articles, clusters, posts, run_log"`.
- `run_cycle()` should open a session, INSERT a `run_log` row with `cycle_id`, `started_at=now(UTC)`, `status='running'`, leave `counts={}`; on cycle end (success or caught exception) UPDATE with `finished_at`, `status`, `counts`. This is the minimum Phase 2 writes; Phase 4+ populate real counts.
- Repository module naming: `src/tech_news_synth/db/{base,session,migrations,hashing,articles,clusters,posts,run_log}.py`. Models live alongside repos or in a single `db/models.py` — Claude picks.
- For the transactional test fixture: `@pytest.fixture def db_session(engine): conn = engine.connect(); trans = conn.begin(); session = Session(bind=conn); yield session; session.close(); trans.rollback(); conn.close()`. Requires a live postgres for the test run; CI via compose or testcontainers.

</specifics>

<deferred>
## Deferred Ideas

- **Active article retention / cleanup job** — revisit when DB size > 10GB or ingest rate increases. Config knob `ARTICLE_RETENTION_DAYS` not added in v1.
- **pgvector extension** — only if TF-IDF client-side cosine proves too slow at scale (won't happen at 24 rows/48h).
- **pg_partman partitioning** — defer until row counts justify it (likely never in v1).
- **Separate `migrate` init container** — valid pattern but adds compose complexity; revisit if startup ordering or multi-service needs emerge.
- **Soft-delete / `deleted_at` columns** — no use case in v1; hard delete is simpler.
- **Read replicas / connection-pool tuning** — single-worker single-host, defaults suffice.
- **UUIDv7 migration** — stick with bigserial; revisit only if cross-system ID uniqueness becomes a need.

</deferred>
