## Summary

Phase 2 delivers four tables (`articles`, `clusters`, `posts`, `run_log`) managed by Alembic with SA 2.0 `Mapped[...]` models and idempotent `ON CONFLICT DO NOTHING` upserts. All infrastructure (postgres service, `Settings.database_url`, structlog, ULID cycle_id, UTC invariants) is already in place from Phase 1 — this phase adds `src/tech_news_synth/db/` with `base.py`, `models.py`, `session.py`, `migrations.py`, `hashing.py`, and repository modules, plus an `alembic/` tree at repo root. Migration runs programmatically from `__main__.py` before `build_scheduler()` (D-01).

**Primary recommendation:** Use SA 2.0 `DeclarativeBase` + `Mapped[...]` with Postgres-specific imports (`sqlalchemy.dialects.postgresql.{JSONB, BYTEA, ARRAY, insert}`). Run `alembic upgrade head` programmatically via `alembic.command.upgrade(Config, "head")` from `run_migrations()`. Use transactional-rollback pytest fixture against compose postgres (not testcontainers) — one live DB, nested `begin_nested()` for commit isolation.

## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01:** `alembic upgrade head` runs in the app container's entrypoint, before `scheduler.start()`. Implementation: extend `src/tech_news_synth/__main__.py` to call a `run_migrations()` helper (from a new `src/tech_news_synth/db/migrations.py`) after `load_settings()` and before `build_scheduler()`. On migration failure, propagate the exception so the container exits non-zero and compose surfaces the error.
- **D-02:** Autogenerate + post-edit migration style. `alembic revision --autogenerate -m "..."` after model changes; human reviews the generated Python and edits to force `TIMESTAMPTZ` (autogenerate emits `DateTime(timezone=True)` which is correct in SA 2.0 but verify), annotate `BYTEA` (`sa.LargeBinary()`) and `JSONB` (`postgresql.JSONB`) columns, add partial/expression indexes if needed, confirm `ON CONFLICT DO NOTHING` lives in upsert helpers, not in the migration.
- **D-03:** Compose `depends_on: service_healthy` is the sole DB-readiness gate. No tenacity retry around alembic or session creation in v1. If postgres is down, compose already blocks app startup; if it flaps mid-cycle, the cycle fails gracefully via INFRA-08 exception isolation and the scheduler retries on the next tick.
- **D-04:** `bigserial` (auto-increment `bigint`) for `articles.id`, `clusters.id`, `posts.id`. Small (8 bytes), Postgres-native, fast FKs, monotonic within each table.
- **D-05:** `run_log.cycle_id` is the natural PK, `TEXT` column storing the 26-char ULID already generated by `tech_news_synth.ids.new_cycle_id()`. No surrogate `run_log.id`. `clusters.cycle_id` and `posts.cycle_id` use `TEXT` + FK to `run_log.cycle_id`.
- **D-06:** `article_hash = SHA256(canonical_url)` (overrides initial SHA1 recommendation). Stored as `CHAR(64)` with `UNIQUE` index. Canonicalization (strip utm_*, lowercase scheme/host, drop fragment, sort query params) is a pure-function helper `tech_news_synth.db.hashing.canonicalize_url(url) -> str` + `article_hash(url) -> str`. Phase 4 ingest uses it; Phase 2 only delivers the helper + test.
- **D-07:** `posts.theme_centroid` is `BYTEA` holding a dense numpy `float32` vector via `np.asarray(x, dtype=np.float32).tobytes()`. Read-back with `np.frombuffer(blob, dtype=np.float32)`. Column is nullable (populated by Phase 5/7, not this phase). No pgvector extension; no sparse TEXT[] representation.
- **D-08:** Passive floor — no active cleanup in v1. STORE-06's "≥14 days of articles" is trivially satisfied because nothing deletes. Volume budget: ~500 articles/day × 365 days ≈ 180k rows/year. No `retention_policy` column, no soft-delete flag.

### Claude's Discretion
- SA 2.0 `DeclarativeBase` subclass layout (single `Base` in `db/base.py` — **recommended** for simplicity).
- Session lifecycle: `sessionmaker` + context manager per `run_cycle()` call; single connection per cycle. Connection pool: default SA `QueuePool` size=5 (single-worker, plenty of headroom).
- Repository helper shape — module-level functions (`articles.upsert_batch(session, rows)`) preferred over classes (matches Phase 1 pure-function style).
- Test DB strategy: **transactional rollback fixture** against compose postgres during CI/local (recommended over testcontainers — simpler, reuses existing compose service).
- Alembic `env.py` online-migration setup (typical boilerplate; use SA 2.0 + psycopg3 DSN from settings).
- `run_log.counts` column type — `JSONB` (flexible, schema evolves in Phase 4/5).
- `posts.status` enum — Postgres `CHECK` constraint with `{pending, posted, failed, dry_run}` (easier to evolve via migration than a pg ENUM type).

### Deferred Ideas (OUT OF SCOPE)
- Active article retention / cleanup job (`ARTICLE_RETENTION_DAYS`) — revisit when DB size > 10GB.
- pgvector extension — only if client-side cosine proves too slow at scale.
- pg_partman partitioning — defer until row counts justify.
- Separate `migrate` init container — valid pattern but adds compose complexity.
- Soft-delete / `deleted_at` columns — no v1 use case.
- Read replicas / connection-pool tuning — single-worker single-host.
- UUIDv7 migration — stick with bigserial.

### Pattern 1: SQLAlchemy 2.0 Typed Declarative Base

**What:** Single `DeclarativeBase` subclass in `db/base.py`; all models inherit. Imports Postgres-specific types where needed.

**When to use:** Every table in this project.

**Example:**

```python
# src/tech_news_synth/db/base.py
# Source: https://docs.sqlalchemy.org/en/20/orm/declarative_tables.html
from sqlalchemy.orm import DeclarativeBase

class Base(DeclarativeBase):
    """Shared metadata for all ORM models. target_metadata for alembic env.py."""
    pass
```

```python
# src/tech_news_synth/db/models.py
from __future__ import annotations
from datetime import datetime
from decimal import Decimal
from typing import Any

from sqlalchemy import (
    BigInteger, CheckConstraint, DateTime, ForeignKey, Index, LargeBinary,
    Numeric, String, Text, func,
)
from sqlalchemy.dialects.postgresql import ARRAY, JSONB
from sqlalchemy.orm import Mapped, mapped_column, relationship

from tech_news_synth.db.base import Base


class RunLog(Base):
    __tablename__ = "run_log"

    cycle_id: Mapped[str] = mapped_column(Text, primary_key=True)  # D-05: ULID natural PK
    started_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), nullable=False, server_default=func.now()
    )
    finished_at: Mapped[datetime | None] = mapped_column(DateTime(timezone=True))
    status: Mapped[str] = mapped_column(Text, nullable=False)  # e.g. running|ok|error|paused
    counts: Mapped[dict[str, Any]] = mapped_column(JSONB, nullable=False, server_default="{}")
    notes: Mapped[str | None] = mapped_column(Text)


class Article(Base):
    __tablename__ = "articles"

    id: Mapped[int] = mapped_column(BigInteger, primary_key=True)  # D-04: bigserial
    source: Mapped[str] = mapped_column(Text, nullable=False)
    url: Mapped[str] = mapped_column(Text, nullable=False)
    canonical_url: Mapped[str] = mapped_column(Text, nullable=False)
    title: Mapped[str] = mapped_column(Text, nullable=False)
    summary: Mapped[str | None] = mapped_column(Text)
    published_at: Mapped[datetime | None] = mapped_column(DateTime(timezone=True))
    fetched_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), nullable=False, server_default=func.now()
    )
    article_hash: Mapped[str] = mapped_column(String(64), nullable=False, unique=True)  # D-06: SHA256 hex
    etag: Mapped[str | None] = mapped_column(Text)
    last_modified: Mapped[str | None] = mapped_column(Text)

    __table_args__ = (
        Index("ix_articles_published_at", "published_at"),
        Index("ix_articles_source_fetched_at", "source", "fetched_at"),
    )


class Cluster(Base):
    __tablename__ = "clusters"

    id: Mapped[int] = mapped_column(BigInteger, primary_key=True)  # D-04
    cycle_id: Mapped[str] = mapped_column(
        Text, ForeignKey("run_log.cycle_id", ondelete="CASCADE"), nullable=False
    )
    member_article_ids: Mapped[list[int]] = mapped_column(
        ARRAY(BigInteger), nullable=False, server_default="{}"
    )
    centroid_terms: Mapped[dict[str, Any]] = mapped_column(
        JSONB, nullable=False, server_default="{}"
    )
    chosen: Mapped[bool] = mapped_column(nullable=False, server_default="false")
    coverage_score: Mapped[float | None] = mapped_column()  # double precision
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), nullable=False, server_default=func.now()
    )

    __table_args__ = (Index("ix_clusters_cycle_id", "cycle_id"),)


class Post(Base):
    __tablename__ = "posts"

    id: Mapped[int] = mapped_column(BigInteger, primary_key=True)  # D-04
    cycle_id: Mapped[str] = mapped_column(
        Text, ForeignKey("run_log.cycle_id", ondelete="CASCADE"), nullable=False
    )
    cluster_id: Mapped[int | None] = mapped_column(
        BigInteger, ForeignKey("clusters.id", ondelete="SET NULL")
    )
    theme_centroid: Mapped[bytes | None] = mapped_column(LargeBinary)  # D-07: BYTEA (numpy float32 bytes)
    status: Mapped[str] = mapped_column(Text, nullable=False)
    tweet_id: Mapped[str | None] = mapped_column(Text)
    cost_usd: Mapped[Decimal | None] = mapped_column(Numeric(10, 6))
    synthesized_text: Mapped[str | None] = mapped_column(Text)
    hashtags: Mapped[list[str]] = mapped_column(
        ARRAY(Text), nullable=False, server_default="{}"
    )
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), nullable=False, server_default=func.now()
    )
    posted_at: Mapped[datetime | None] = mapped_column(DateTime(timezone=True))
    error_detail: Mapped[str | None] = mapped_column(Text)

    __table_args__ = (
        CheckConstraint(
            "status IN ('pending', 'posted', 'failed', 'dry_run')",
            name="ck_posts_status",
        ),
        Index("ix_posts_cycle_id", "cycle_id"),
        Index("ix_posts_posted_at", "posted_at"),
        Index(
            "ix_posts_posted_at_48h",
            "posted_at",
            postgresql_where=(Post.posted_at.isnot(None)),
        ),
    )
```

**Notes on typing:**
- `Mapped[...]` with SA 2.0 derives column type from the Python annotation when unambiguous. For ambiguous types (BigInteger vs default Integer, Text vs String, BYTEA vs generic bytes) pass `mapped_column(LargeBinary)` / `mapped_column(BigInteger)` etc. explicitly — autogenerate otherwise picks `INTEGER` / `VARCHAR` which we do NOT want. [CITED: https://docs.sqlalchemy.org/en/20/orm/declarative_tables.html]
- `DateTime(timezone=True)` maps to Postgres `TIMESTAMPTZ`. Confirmed in SA 2.0 dialect. [CITED: https://docs.sqlalchemy.org/en/20/dialects/postgresql.html#datetime-types]
- `server_default=func.now()` emits `DEFAULT NOW()` which, on a `TIMESTAMPTZ` column, is UTC when the session is UTC (Phase 1 enforces). [VERIFIED: Postgres 16 docs — `now()` returns `timestamp with time zone`.]

### Pattern 2: Alembic env.py Reading from Settings

**What:** `env.py` imports `tech_news_synth.config.load_settings()` and `tech_news_synth.db.base.Base`, passes `target_metadata = Base.metadata`, and reads DSN from settings — **never** from `alembic.ini`.

**When to use:** Both offline (`alembic upgrade --sql`) and online (`command.upgrade(cfg, "head")`) paths.

**Example:**

```python
# alembic/env.py
# Source: https://alembic.sqlalchemy.org/en/latest/autogenerate.html + project's pydantic-settings
from __future__ import annotations
from logging.config import fileConfig

from alembic import context
from sqlalchemy import engine_from_config, pool

from tech_news_synth.config import load_settings
from tech_news_synth.db.base import Base
from tech_news_synth.db import models  # noqa: F401 — register models on Base.metadata

config = context.config
if config.config_file_name is not None:
    fileConfig(config.config_file_name)

# Inject DSN from pydantic-settings (SecretStr materialized exactly once here)
_settings = load_settings()
config.set_main_option("sqlalchemy.url", _settings.database_url)

target_metadata = Base.metadata


def run_migrations_offline() -> None:
    url = config.get_main_option("sqlalchemy.url")
    context.configure(
        url=url,
        target_metadata=target_metadata,
        literal_binds=True,
        dialect_opts={"paramstyle": "named"},
        compare_type=True,  # so TIMESTAMPTZ vs TIMESTAMP drift is detected
    )
    with context.begin_transaction():
        context.run_migrations()


def run_migrations_online() -> None:
    connectable = engine_from_config(
        config.get_section(config.config_ini_section, {}),
        prefix="sqlalchemy.",
        poolclass=pool.NullPool,
    )
    with connectable.connect() as connection:
        context.configure(
            connection=connection,
            target_metadata=target_metadata,
            compare_type=True,
            compare_server_default=True,
        )
        with context.begin_transaction():
            context.run_migrations()


if context.is_offline_mode():
    run_migrations_offline()
else:
    run_migrations_online()
```

**What:** Python-callable alembic runner invoked from `__main__.py` before `build_scheduler()`.

```python
from __future__ import annotations
from pathlib import Path

from alembic import command
from alembic.config import Config

from tech_news_synth.logging import get_logger

# Repo-relative (container path is /app — alembic/ copied there by Dockerfile)

def run_migrations() -> None:

    log.info("alembic_upgrade_start", target="head")
    command.upgrade(cfg, "head")
    log.info("alembic_upgrade_done")
```

### Pattern 4: `ON CONFLICT DO NOTHING` Upsert

**What:** Dialect-specific INSERT with `on_conflict_do_nothing` — no exception on duplicate `article_hash`, no row returned.

**When to use:** Article ingestion (Phase 4 uses this; Phase 2 delivers the helper + test).

**Example:**

```python
# src/tech_news_synth/db/articles.py
# Source: https://docs.sqlalchemy.org/en/20/dialects/postgresql.html#insert-on-conflict-upsert
from __future__ import annotations
from collections.abc import Sequence
from typing import TypedDict

from sqlalchemy import select
from sqlalchemy.dialects.postgresql import insert as pg_insert
from sqlalchemy.orm import Session

from tech_news_synth.db.models import Article


class ArticleRow(TypedDict):
    source: str
    url: str
    canonical_url: str
    title: str
    summary: str | None
    published_at: "datetime | None"
    article_hash: str
    etag: str | None
    last_modified: str | None


def upsert_batch(session: Session, rows: Sequence[ArticleRow]) -> int:
    """Insert-or-skip on article_hash conflict. Returns inserted row count."""
    if not rows:
        return 0
    stmt = (
        pg_insert(Article)
        .values(list(rows))
        .on_conflict_do_nothing(index_elements=["article_hash"])
    )
    result = session.execute(stmt)
    # rowcount reflects rows actually inserted (psycopg3 reports accurate count on ON CONFLICT)
    return result.rowcount or 0


def get_by_hash(session: Session, article_hash: str) -> Article | None:
    return session.execute(
        select(Article).where(Article.article_hash == article_hash)
    ).scalar_one_or_none()
```

**Returning behavior on conflict:** `on_conflict_do_nothing` does NOT return the existing row (PG behavior — `RETURNING` emits nothing for skipped rows). If you need the existing row, do a separate `SELECT` or use `on_conflict_do_update(...).returning(...)`. For article dedup we only care about "is it new?" — `rowcount` suffices. [CITED: https://www.postgresql.org/docs/16/sql-insert.html#SQL-ON-CONFLICT]

```python
        settings.database_url,  # postgresql+psycopg://... (Phase 1)
        pool_pre_ping=True,     # catches dead-connection-in-pool after postgres restart
        pool_size=5,            # default; single-worker has plenty of headroom
        future=True,            # SA 2.0 idiom (already the default in 2.0, explicit for clarity)
    )
    _SessionLocal = sessionmaker(bind=_engine, expire_on_commit=False, class_=Session)


@contextmanager
def get_session() -> Iterator[Session]:
    """Session context manager for one unit of work. Commits on success, rolls back on exception.

    Usage:
        with get_session() as session:
            articles.upsert_batch(session, rows)
            # session.commit() called by ctx exit
    """
    if _SessionLocal is None:
        raise RuntimeError("init_engine() must be called before get_session()")
    session = _SessionLocal()
    try:
        yield session
        session.commit()
    except Exception:
        session.rollback()
        raise
    finally:
        session.close()
```

**`expire_on_commit=False`:** Allows accessing ORM attributes after commit without triggering a refresh. Standard for modern SA 2.0 apps. [CITED: https://docs.sqlalchemy.org/en/20/orm/session_api.html#sqlalchemy.orm.sessionmaker]

### Pattern 6: `run_cycle()` Integration (Phase 2 Skeleton)

**What:** `scheduler.run_cycle()` opens a session, writes a `run_log` row at cycle start, updates at cycle end. Phase 4+ populate real counts.

```python
# src/tech_news_synth/scheduler.py (amended)
from tech_news_synth.db.session import get_session
from tech_news_synth.db import run_log as run_log_repo

def run_cycle(settings: Settings) -> None:
    cycle_id = new_cycle_id()
    bind_contextvars(cycle_id=cycle_id, dry_run=bool(settings.dry_run))
    try:
        paused, reason = is_paused(settings)
        if paused:
            log.info("cycle_skipped", status="paused", paused_by=reason)
            return

        log.info("cycle_start", interval_hours=settings.interval_hours)

        with get_session() as session:
            run_log_repo.start(session, cycle_id=cycle_id)

        status = "ok"
        try:
            _run_cycle_body(settings)   # Phase 4+ body here
        except Exception:
            log.exception("cycle_error")
            status = "error"
        finally:
            with get_session() as session:
                run_log_repo.finish(session, cycle_id=cycle_id, status=status, counts={})

        log.info("cycle_end", status=status)
    finally:
        clear_contextvars()
```

**Two separate sessions** (start + finish) — simpler than holding a session across the entire cycle; downside is `run_log` row is committed at start so a crash mid-cycle leaves `status='running'` with no `finished_at`. Acceptable for v1; operator can query `WHERE status='running' AND started_at < NOW() - INTERVAL '1 hour'` to find zombies.

### Anti-Patterns to Avoid

- **`sqlalchemy.url` in `alembic.ini`:** Leaks DB password via git. Always set it in `env.py` from pydantic-settings.
- **Naive `datetime` in model defaults:** Use `server_default=func.now()` (Postgres-side UTC) or `default=lambda: datetime.now(UTC)` (Python-side) — never `default=datetime.utcnow` (deprecated and naive).
- **`postgresql+psycopg2://` DSN:** We install psycopg3 only. Using the psycopg2 dialect will fail at connect time.
- **Mixing `Integer` and `BigInteger`:** FKs must match. `articles.id` is `BigInteger`; any column referencing it must also be `BigInteger` (e.g. `member_article_ids: ARRAY(BigInteger)`).
- **CHECK constraint via Python enum only:** SA's `Enum` type creates a pg ENUM, which is hard to ALTER. Use a `CHECK` constraint with a string column (per CONTEXT.md discretion).
- **Storing centroid as JSONB or TEXT[]:** Inflates size ~4-8×. `BYTEA` of `np.float32.tobytes()` is compact and round-trips cleanly.
- **`datetime.utcnow()`:** Deprecated in Python 3.12, returns naive. Always `datetime.now(timezone.utc)` (ruff DTZ already enforces).

## Common Pitfalls

### Pitfall 1: Autogenerate misses `TIMESTAMPTZ` vs `TIMESTAMP`

**What goes wrong:** Alembic autogenerate may emit `sa.DateTime()` instead of `sa.DateTime(timezone=True)` depending on how models declare columns.
**Why it happens:** Default SA `DateTime` has `timezone=False`. Must explicitly set it on every datetime column.
**How to avoid:** In models use `DateTime(timezone=True)` everywhere. In `env.py` set `compare_type=True` so type drift is detected. Review autogenerated migration line-by-line for `sa.DateTime(timezone=True)` (D-02).
**Warning signs:** `psycopg.errors.DatatypeMismatch` at insert time, or naive `datetime` objects coming back from queries.

### Pitfall 2: `CheckConstraint` dropped by autogenerate

**What goes wrong:** On subsequent autogenerate runs, `compare_type=True` may not always detect CHECK constraint changes, or the human-authored constraint gets overwritten.
**Why it happens:** Alembic's CHECK constraint detection is limited (`compare_server_default=True` helps but not for CHECKs).
**How to avoid:** Place CHECK constraints in `__table_args__` (as in models example above). In first migration (hand-edited per D-02), the CHECK is explicit. On future autogenerate runs, diff carefully — do not blindly accept deletions of CHECK constraints.

### Pitfall 3: `server_default="{}"` for JSONB raises on empty dict

**What goes wrong:** `server_default="{}"` is literal PG — for JSONB it renders as `'{}'::jsonb` correctly, but for SA to emit it the string `"{}"` must be wrapped in `sa.text("'{}'::jsonb")` in some versions.
**Why it happens:** SA interprets plain `"{}"` as identifier by default for some column types.
**How to avoid:** Use `server_default=sa.text("'{}'::jsonb")` explicitly in migration file. For `ARRAY` columns, `server_default=sa.text("'{}'::bigint[]")` or `"'{}'::text[]"`. Test the migration against a real postgres container.
**Warning signs:** Migration succeeds but INSERT fails with "column violates NOT NULL" because default didn't apply.

### Pitfall 4: Connection string in Alembic stdout

**What goes wrong:** Default alembic logging prints `sqlalchemy.url` in INFO lines — leaking the DB password to stdout (and structlog JSON sink).
**Why it happens:** `alembic.runtime.migration` and `alembic.ddl.postgresql` loggers are verbose by default.
**How to avoid:** In `alembic.ini` `[loggers]` section set `level = WARN` for `sqlalchemy.engine` and `alembic.runtime.migration`. Never pass DSN via `-x` CLI args (logged in shell history). Test: `docker compose up` + `grep -i 'postgres_password\|@postgres:5432' /data/logs/app.jsonl` → zero hits.
**Warning signs:** Password substring appears in `logs/app.jsonl`.

### Pitfall 5: FK order in first migration

**What goes wrong:** `clusters.cycle_id` FK to `run_log.cycle_id` — if `clusters` table is created before `run_log` in the migration, DDL fails.
**Why it happens:** Autogenerate usually orders correctly by dependency, but occasionally ties are ambiguous.
**How to avoid:** Hand-review the generated migration's `op.create_table(...)` order: `run_log` → `articles` → `clusters` → `posts`. `downgrade()` drops in reverse order.

### Pitfall 6: `pool_pre_ping` cost at very high QPS

**What goes wrong:** `pool_pre_ping=True` adds a `SELECT 1` before each checkout. At thousands of QPS this matters.
**Why it happens:** Each ping is a roundtrip.
**How to avoid:** At our scale (~12 cycles/day × ~5 connections each), the cost is negligible. Keep it on — the resilience to postgres restarts is worth it.
**Warning signs:** Not applicable at our scale.

### Pitfall 7: Alembic command.upgrade() holds a connection open on failure

**What goes wrong:** If `command.upgrade(cfg, "head")` raises, the alembic-created engine may not be disposed; in a short-lived script this is harmless but in a long-lived container it would leak.
**How to avoid:** `command.upgrade` uses `env.py`'s `run_migrations_online` which creates a `NullPool` engine (no pooling) — connection is closed by the `with connectable.connect()` block on exception. No action needed.

## Code Examples

### URL canonicalization (D-06)

```python
# src/tech_news_synth/db/hashing.py
# Source: https://docs.python.org/3/library/urllib.parse.html
from __future__ import annotations
import hashlib
from urllib.parse import parse_qsl, urlencode, urlsplit, urlunsplit

_STRIP_QS_PREFIXES = ("utm_",)
_STRIP_QS_EXACT = frozenset({"gclid", "fbclid", "mc_cid", "mc_eid"})


def canonicalize_url(url: str) -> str:
    """Pure function: lowercase scheme+host, drop fragment, strip tracking params, sort query.

    Not responsible for:
      - punycode/IDNA conversion (assumes input is already ASCII or valid IRI)
      - default-port normalization (Phase 2 scope keeps ports as-is)
      - trailing-slash normalization (deferred — different paths are different resources)
    """
    parts = urlsplit(url.strip())
    scheme = parts.scheme.lower()
    netloc = parts.netloc.lower()
    path = parts.path  # case-sensitive per RFC 3986 §6.2.2.1
    # Strip tracking params + sort alphabetically
    kept = [
        (k, v) for k, v in parse_qsl(parts.query, keep_blank_values=True)
        if not k.startswith(_STRIP_QS_PREFIXES) and k not in _STRIP_QS_EXACT
    ]
    kept.sort(key=lambda kv: (kv[0], kv[1]))
    query = urlencode(kept)
    # Drop fragment (never identifies a distinct article)
    return urlunsplit((scheme, netloc, path, query, ""))


def article_hash(url: str) -> str:
    """SHA256 hex (64 chars) of the canonicalized URL. D-06."""
    canonical = canonicalize_url(url)
    return hashlib.sha256(canonical.encode("utf-8")).hexdigest()
```

### numpy centroid BYTEA roundtrip (D-07)

```python
# src/tech_news_synth/db/posts.py — centroid helpers
from __future__ import annotations
import numpy as np
from numpy.typing import NDArray


def centroid_to_bytes(vec: NDArray[np.floating]) -> bytes:
    """Serialize a numpy vector as float32 little-endian bytes for BYTEA storage.

    Deliberately coerces dtype — callers may pass float64 (sklearn default).
    """
    arr = np.asarray(vec, dtype=np.float32)
    if arr.ndim != 1:
        raise ValueError(f"centroid must be 1D, got shape {arr.shape}")
    return arr.tobytes()  # C order, little-endian on all supported platforms


def centroid_from_bytes(blob: bytes) -> NDArray[np.float32]:
    """Inverse of centroid_to_bytes. Caller knows dimension from context."""
    return np.frombuffer(blob, dtype=np.float32)
```

**Dimension note:** CLUSTER-02 uses char n-grams 3-5 → TF-IDF matrix has ~10k features (one per observed n-gram). But Phase 2 stores a **centroid** (mean vector of a cluster), which has the **same dimension as the feature space at cluster time**. That dimension is NOT fixed across cycles because TF-IDF vocabulary is fit per-cycle on the 6h window. **Implication for Phase 5/7 anti-repetition:** the 48h centroid comparison needs vocabulary alignment — Phase 5 must decide whether to (a) store top-K weighted terms as JSONB alongside the float32 blob and recompute cosine on intersection, or (b) project into a fixed-dim space (e.g., HashingVectorizer). **Phase 2 scope:** store the bytes, document the ambiguity, defer the comparison logic to Phase 5.

## Open Questions

1. **Should `configure_logging()` move to `_dispatch_scheduler()` or stay in `scheduler.run()`?**
   - What we know: Phase 1 calls `configure_logging()` inside `scheduler.run()`. Phase 2 wants alembic output captured by structlog, which requires logging configured **before** `run_migrations()`.
   - What's unclear: whether moving the call breaks Phase 1's test fixtures that call `scheduler.run()` directly.
   - Recommendation: **Move** `configure_logging(settings)` to `_dispatch_scheduler()` (before `run_migrations()`), and make `scheduler.run()` idempotent-tolerant (structlog's `configure()` is idempotent — safe to call again). Planner should check Phase 1 tests for fragility.

2. **Should the first migration's `downgrade()` be real or no-op?**
   - What we know: STORE-01 success criterion says "downgrade path exists for at least the latest migration."
   - What's unclear: semantic — is a `drop_table` downgrade enough, or does the criterion want a meaningful revert (which makes no sense for the very first migration)?
   - Recommendation: implement `downgrade()` as `drop_table(...)` in reverse order. Document in runbook that downgrading the initial migration wipes all data. Test verifies `downgrade()` reverses `upgrade()` byte-for-byte via `Base.metadata` comparison.

3. **Where does `init_engine(settings)` get called?**
   - What we know: `get_session()` requires `_SessionLocal` initialized.
   - Recommendation: call `init_engine(settings)` in `_dispatch_scheduler()` right after `configure_logging(settings)`, before `run_migrations()`. Migrations use alembic's own engine, so order doesn't matter for correctness — but consistency matters. Planner confirms.

4. **Is there a need for a `JSONB`-backed `run_log.counts` schema-on-write validator?**
   - What we know: `counts` is JSONB, populated by Phase 4+ with per-source fetch counts + cluster count.
   - Recommendation: Phase 2 stores opaque `dict[str, Any]`; add pydantic schema in Phase 4 when the shape is known. No validator in Phase 2.
