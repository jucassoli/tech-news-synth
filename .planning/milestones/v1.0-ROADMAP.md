## Outcome

Python agent that every 2h pulls tech news from 5 public feeds (TechCrunch/Verge/Ars RSS + Hacker News Firebase + Reddit r/technology JSON), clusters by TF-IDF title similarity, synthesizes the highest-coverage cluster into one PT-BR tweet via Claude Haiku 4.5, and publishes to @ByteRelevant with hard guardrails on daily cap (12/day), monthly cost ($30), rate limits, and idempotency. Fully containerized, single-operator-deployable via `docker compose up -d`.

**Operator sign-off:** Phase 8 Parts A (CLI smoke) + B (48h DRY_RUN soak) + C (live cutover + 24h verify) all approved. Agent is ticking in production.

## Key Accomplishments

1. **Pure-core phase architecture** — each phase ships a composable `run_<phase>()` function (pure) + an orchestrator wiring into `scheduler.run_cycle`. 8 phases, zero circular deps, every phase has its own plan→execute→verify cycle.

2. **Cost-envelope validated empirically** — Phase 3 GO at $0.03/post × 12/day × 30 days = **$10.80/mo**, 2× under the $20-50 envelope. Phase 8 cutover_verify.py confirms production costs match the baseline within 2× tolerance.

3. **Semantic 48h anti-repeat** — Phase 5 re-fits TF-IDF on combined corpus (current cluster + last-48h post source texts) at compare time, yielding vocabulary-stable cosine similarity despite per-cycle ephemeral vectorizers. Avoids the naive cross-cycle centroid comparison trap.

4. **Idempotent publishing despite no-idempotency-key API** — X v2 `create_tweet` has no idempotency-key header. Phase 7 works around via a 5-minute stale-pending guard that catches orphaned `status='pending'` rows (mid-call crash window) and marks them `failed` before any retry, with a documented operator runbook for rare genuine double-write investigations.

5. **Deterministic clustering + verifiable cadence** — Phase 5 winner selection is fully deterministic (fixtures verified 3× identical runs); Phase 8 48h soak proved ≥24 cycles without unhandled exceptions; operator CLIs (`replay`, `post-now`, `source-health`) let the operator inspect any cycle post-hoc without touching the DB directly.

6. **Bonus defect caught cross-phase** — Phase 7 research uncovered a latent Phase 2 bug in `db.posts.update_posted` that would have overwritten `cost_usd=NULL` on state transition, corrupting the monthly-cost cap. Fixed with regression test before it could manifest in production.

## Lessons

- **Research catches bugs before they ship.** Phase 5 found sklearn's silent `stop_words` ignore with `char_wb`; Phase 7 caught Phase 2's `cost_usd` overwrite; Phase 8 research confirmed where `cycle_summary` must emit to preserve durability invariants.
- **CONTEXT-first planning.** Every phase locks decisions before research; researcher validates, planner honors, verifier goal-checks. No gray areas regressed across phases.
- **Operator involvement at checkpoints.** 6 of 8 phases had a `checkpoint:human-verify` task. This caught boot-observability gaps (Phase 5 step 7) that automated tests missed.
- **Pure-function phase architecture scales.** Every phase's `run_<phase>()` is testable in isolation; the scheduler composes them; DRY_RUN threads through `status='dry_run'` without any phase needing explicit dry-run branches.
- **Cost observability wins.** Phase 3's empirical $0.03/post measurement made Phase 7's monthly-cost cap design trivial; Phase 8's `cutover_verify.py` uses the same baseline for production acceptance.
